# 工具链与沙盒设计 (Toolchain Design)

## 1. 现状概述

Nimbus 提供了一套标准化的底层工具能力（Toolchain），这是 Agent 与操作系统、代码库和业务环境交互的唯一桥梁。它被设计为高度抽象、模块化且具备极强的防御性和扩展性。

## 2. 标准工具集 (Standard Toolset)

### 2.1 文件操作 (File Operations)
文件系统是 Agent 感知世界和修改世界的主要手段：
- **`Read` (读取)**: 
  - **核心功能**: 读取文件内容。
  - **防御机制**: 支持 `offset` 和 `limit` 参数，允许 Agent 分片读取大型日志文件或巨大的代码库文件，防止单次读取撑爆 LLM 上下文。
- **`Write` (写入)**: 
  - **核心功能**: 将内容直接写入文件，如果文件不存在会自动创建其父目录。
  - **使用场景**: 创建新文件、重写现有文件、记录临时日志或配置文件。
- **`Edit` (编辑)**: 
  - **核心功能**: 替换文件中的特定文本段落（`old_text` -> `new_text`）。
  - **约束条件**: `old_text` 必须与文件中的内容精确匹配（包括空格和换行），这迫使 LLM 仔细核对代码上下文，减少破坏性修改的风险。
- **`Grep` (搜索)**: 
  - **核心功能**: 在指定文件或目录中搜索文本模式。
  - **返回格式**: 返回匹配的行、文件路径和行号，是快速定位代码逻辑和排查错误日志的利器。

### 2.2 环境执行 (Environment Execution)
- **`Bash` (终端命令执行)**: 
  - **核心功能**: 提供一个受限的终端环境，允许 Agent 执行各种 Linux/Unix 命令（如 `ls`, `cat`, 运行测试脚本、构建项目等）。
  - **防御机制**: 这是最危险也是最容易导致上下文暴涨的工具。Nimbus 强制实施了输出截断（例如最后 2000 行或 50KB）。一旦发生截断，完整的输出会被自动保存到一个临时文件中，并返回给 LLM 该文件的路径，LLM 随后可以使用 `Read` 或 `Grep` 分片或精准检索完整日志。
- **`spawn_agent` (派生子节点)**:
  - **核心功能**: Orchestrator 专用的编排工具。用于拉起一个或多个 Sub-Agent。
  - **输入要求**: 强制要求提供明确的目标（Goal）、相关的代码/文件路径（Context）、期望的输出格式，并分配特定角色（如 `reader` 或 `worker`）。

## 3. 防爆机制与安全护栏 (Safeguards)

长序列处理与不确定的环境执行是 LLM Agent 面临的最大挑战。Nimbus 在 Toolchain 层面内置了多重防爆机制：
1. **输入/输出强约束**: 强制的 `Read` 分片、`Bash` 输出硬截断、`Edit` 的精确匹配要求。
2. **状态隔离 (Isolation)**: Agent 运行在特定的工作目录（Workspace）中，所有持久化的思考记录（Scratchpad）和临时日志都有明确的命名空间和路径约定（如 `.nimbus/sessions/sess_xxx/`）。
3. **隔离的沙盒环境理念 (Sandbox Philosophy)**: 尽管当前 `Bash` 直接执行宿主机命令，但在概念设计上，Sub-Agents 被视为在隔离的沙盒中运行，其能力边界（如网络访问、敏感目录读写）可以通过底层 Toolchain 的封装进行限制。

## 4. 扩展性 (Extensibility)

Nimbus 的 Toolchain 架构允许轻松接入新的外部能力。例如，在“商业智能剪辑”落地时，可以将核心剪辑算法 API 封装为新的 Tool（如 `Video_Clip`, `Audio_Extract`），并将其注册到特定角色（如 `video_worker`）的可用工具列表中，从而无缝扩展 Agent 的业务能力边界。
