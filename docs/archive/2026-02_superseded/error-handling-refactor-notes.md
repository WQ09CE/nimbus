# Error Handling & Memory Refactoring Notes (2026-02-01)

This document records the insights and architectural changes from the deep troubleshooting session regarding Agent "Doom Loops" and Context Loss.

## 1. The "Doom Loop" Incident

**Symptom**: The Agent would successfully read a file (e.g., `vcpu.py`), but in the very next turn, it would act as if the file was never read, attempting to read it again repeatedly until it hallucinated that the file "doesn't exist".

**Root Cause**: **False Positive Failure Detection** leading to **Context Loss**.

1.  **The Trigger**: The source code of `vcpu.py` contained the string `"ENOENT"` (in a comment).
2.  **The Flaw**: The MMU's `_auto_detect_tool_failure` had a loose heuristic for the `Read` tool: `if "ENOENT" in content`.
3.  **The Chain Reaction**:
    *   The system misidentified the *successful* file content as an error message.
    *   It marked the tool result as `discardable`.
    *   The `assemble_context` logic filtered out this "discardable" message to save tokens.
    *   **Result**: The LLM received a context history where it asked to read the file, but the result was missing. It reasonably concluded the action hadn't happened or failed silently, so it tried again.

## 2. Key Insights

### Insight 1: "Gaslighting" the LLM is Dangerous
Hiding failed attempts (Eager Filtering) creates a disconnect between the LLM's actions and the perceived reality.
*   **Old Approach**: Delete failed tool calls from history to save context window.
*   **Consequence**: LLM repeats the same mistake because it has no memory of the failure.
*   **New Approach**: **Lazy Compaction**. Keep all history (including failures) by default. Only filter when absolutely necessary (token budget exceeded).

### Insight 2: Explicit > Implicit
Relying on content matching (heuristics) for control flow is fragile, especially for a Coding Agent that handles arbitrary code/text.
*   **Old Approach**: `if "error" in content.lower()`
*   **Consequence**: Reading a log file or source code containing "error" triggers false failure detection.
*   **New Approach**: Rely on **Structured Signals**.
    *   Python Exceptions -> `Fault` objects.
    *   Standardized `[Error]` prefixes generated by the Kernel Gate.
    *   Exit codes (for Bash).

### Insight 3: ID Collision in Memory Management
We discovered that filtering messages based on a global set of "discardable IDs" is risky.
*   **Scenario**: Tool A fails (ID: 123) -> Retry Tool A succeeds (ID: 123).
*   **Bug**: If the LLM/Adapter reuses the Tool Call ID for the retry, marking ID 123 as discardable hides *both* the failure and the success.
*   **Fix**: Mark specific **Message instances** in the `StackFrame`, not global IDs.

## 3. Architecture Refactoring

We pushed the "Error Standardization" logic down to the **Gate** layer to separate concerns.

### Before
*   **Gate**: Caught exception, returned `Fault` object with raw message.
*   **vCPU**: Checked for `Fault`, manually prepended `"[Error] "` string before adding to memory.
*   **MMU**: Had mixed logic for detecting errors (checking for prefixes AND loose content matching).

### After (Standardized)
*   **Gate (The Customs)**:
    *   Catches Exceptions, Timeouts, Cancellations.
    *   **Responsibilities**: Always returns a `ToolResult` where `output` is formatted as `"[Error] <message>"`.
    *   The `Fault` object is still attached for programmatic access, but `output` is ready for LLM consumption.
*   **vCPU (The Brain)**:
    *   Trusts `result.output` from Gate.
    *   **Error Recovery**: Instead of injecting *new* system messages (which breaks conversation flow), it **enhances** the `ToolResult.output` by appending hints (e.g., `[Error] ... \n\n(Hint: Try ...)`).
*   **MMU (The Memory)**:
    *   **Failure Detection**: Only checks for `[Error]` prefix or `Traceback` patterns. No loose keywords.
    *   **Compaction**: Defaults to keeping everything.

## 4. Design Principles Update

1.  **Lazy Compaction**: Don't optimize context prematurely. Let the LLM see its failures.
2.  **Hint > Hiding**: When helping the LLM recover, append context/hints to the error rather than hiding the error and injecting a system instruction.
3.  **One Gate**: The Kernel Gate is the single source of truth for tool execution status and output formatting.
