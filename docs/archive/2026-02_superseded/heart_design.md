# Nimbus Framework: "Heart" 组件架构设计文档

## 1. 概述
"Heart" 是 Nimbus 框架中的系统级常驻后台守护进程，负责在后台默默维护和优化系统的状态、记忆和健康。它是实现 Agent 长期自主性和自我进化的关键基础设施。

## 2. 核心职责定义

1. **Memory Consolidation (记忆整合)**: 异步处理和压缩长期记忆，将零散的事件提炼为有价值的经验和模式。
2. **Health Monitoring (健康监控)**: 持续监控系统各组件（如 Brain, NIMFS, vCPU 等）的运行状态和资源消耗。
3. **Proactive Alerting (主动告警)**: 在发现异常、资源耗尽或性能下降时，主动向系统或用户发出告警。
4. **Self-Evolution (自我进化)**: 基于长期的运行数据和反馈，自动提炼改进建议，优化系统 Prompt、工具配置或架构模式。

## 3. 关键技术解决方案 (响应 AI 评审反馈)

### 3.1 并发与竞态条件管理
- **策略**: 采用多版本并发控制 (MVCC) 或细粒度的读写锁机制，确保 Heart 在后台处理记忆时不会阻塞前台 Brain 的实时读写。
- **Idle 状态运行**: 优先在系统空闲 (Idle) 时段执行重度计算任务（如记忆压缩），减少对主业务的资源抢占。

### 3.2 异步 Inbox 通信机制
- **策略**: Brain 与 Heart 之间通过严格的异步 Inbox 队列进行解耦通信。Brain 仅将事件（如对话记录、错误日志）推入 Inbox，Heart 负责拉取和处理。
- **优势**: 避免 Brain 阻塞，提高系统响应速度。

### 3.3 成本控制与增量处理
- **Dirty Bit 机制**: 为记忆条目引入 Dirty Bit 标记。Heart 仅处理自上次处理以来发生变更或新增的条目。
- **增量处理**: 避免全量扫描，显著降低大语言模型 (LLM) 调用的 Token 消耗和计算成本。

### 3.4 事件驱动的可靠性语义
- **幂等性**: Heart 处理事件的操作必须是幂等的，确保重复处理不会导致数据错误。
- **重试与 DLQ (死信队列)**: 失败的事件处理任务将进入重试策略。达到最大重试次数后，移入 DLQ 供人工审查或后续排查。

### 3.5 记忆生命周期 Schema
定义清晰的记忆提炼层级：
1. **Raw Event (原始事件)**: 原始的交互日志、工具调用记录、错误信息。
2. **Case (具体案例)**: 经过初步聚类和总结的具体场景或问题解决过程。
3. **Pattern (模式规律)**: 从多个类似 Case 中提取的通用模式、常见坑点或最佳实践。
4. **Principle (核心原则)**: 最高的抽象层级，转化为系统的核心指导原则（可直接注入 System Prompt）。

### 3.6 健康监控的 SLOs 与阈值
- 定义具体的服务等级目标 (SLOs)，如：
  - Inbox 队列积压延迟 < 5 分钟
  - 记忆压缩成功率 > 99%
  - vCPU 错误率 < 1%
- 设定明确的告警阈值，触发 Proactive Alerting。

### 3.7 安全的自我进化流水线
为了防止 "污染" 导致系统崩溃，自我进化必须经过严格的流水线：
1. **候选 (Candidate)**: Heart 提出改进建议（如修改某工具的描述）。
2. **回放 (Replay)**: 在沙盒环境中，使用历史 Case 回放验证该修改是否能带来正面收益且无回归 (Regression)。
3. **审批 (Approval)**: 关键变更需人工确认或经过严格的自动化断言。
4. **灰度 (Canary)**: 在部分任务中灰度应用，监控无异常后全量推广。

## 4. 实施路径规划

建议采用 **混合模式 (Hybrid Mode)** 逐步演进，优先实现规则引擎：

### Phase 1: 基础心跳与规则引擎 (Weeks 1-2)
- 实现 Heart 守护进程的基础生命周期管理。
- 搭建异步 Inbox 队列。
- **优先实现**: 基于简单规则的 Health Monitoring 和 Proactive Alerting（如队列长度监控、错误率阈值告警）。

### Phase 2: 记忆增量压缩 (Weeks 3-4)
- 引入 Dirty Bit 机制。
- 实现 Raw Event -> Case 的初步压缩逻辑。
- 引入幂等和 DLQ 机制保障可靠性。

### Phase 3: 高级提炼与并发控制 (Weeks 5-6)
- 实现 Case -> Pattern -> Principle 的深度提炼。
- 完善 MVCC 或读写锁，确保高并发下的数据一致性。
- 优化 Idle 触发机制。

### Phase 4: 自我进化流水线 (Weeks 7+)
- 搭建沙盒回放环境。
- 实现候选、回放、审批、灰度的完整进化闭环。
